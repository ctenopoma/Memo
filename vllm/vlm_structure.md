# LLMとVLMのモデル構造と処理プロセスの違い

## 1. LLMとVLMの根本的な違い

最大の違いは、**「視覚データ（ピクセル）」を「言語データ（単語/トークン）」と同じ土俵で計算できる仕組みを持っているか**にあります。

* **LLM（大規模言語モデル）**
  * **入力:** テキストのみ
  * **処理:** テキストを「トークン」に分割し、Transformer層で文脈を解釈して次の言葉を予測する。人間の「言語野」に特化した構造。

* **VLM（視覚言語モデル）**
  * **入力:** テキスト ＋ 画像（動画）
  * **処理:** 「画像のピクセル配列」と「テキストのトークン」を、モデル内部で同じ意味を持つ共通のベクトル空間に統合して処理する。

---

## 2. VLMの基本構造と `mmproj` の役割

VLMは基本的に以下の3つのコンポーネントで構成されます。

1. **Vision Encoder（視覚エンコーダ / ViT等）**: 「目」の役割。画像をパッチに分割し、視覚的な特徴を抽出する。
2. **Projector（プロジェクター / `mmproj` 等）**: 「翻訳機」の役割。Vision Encoderの出力をLLMが理解できる形式に変換する。
3. **LLM本体**: 「脳」の役割。テキストと「翻訳された画像データ」を受け取り、推論して回答を生成する。

### なぜ `mmproj`（プロジェクター）が必要なのか？
「画像の表現空間」と「言語の表現空間」の次元と意味をすり合わせるためです。視覚的な特徴ベクトルを、LLMが直接扱える**「視覚トークン（Visual Tokens）」**に翻訳しないと、LLMにとって画像データは意味不明なノイズになってしまいます。

---

## 3. モデル構造の進化（Llama 3.2 Vision から Qwen 3.5 へ）

### 第1世代：モジュール結合型（Llama 3.2 Vision, Qwen 2-VLなど）
既存のLLMに、後付けで画像処理モジュールを繋ぎ合わせた構造。

* **Llama 3.2 Vision**: LLMの層の間に「Cross-Attention Vision Adapter」を差し込み、必要な時に画像特徴を参照する仕組み。
* **Qwen 2-VL**: 独自の「動的解像度（画像を歪めず元の比率でパッチ化）」とプロジェクターを組み合わせ、高いOCR性能を実現。

### 第2世代：ネイティブ・マルチモーダル型（Qwen 3.5など）
初期の事前学習段階から、テキストと画像を「ひとつの共通言語」として同時に学習（Early Fusion）させた統合型アーキテクチャ。

* **深い融合（DeepStack）**: 視覚処理層とLLMの推論層が深部で直接絡み合う。
* **効率的な推論（MoE / ハイブリッド・アテンション）**: Gated DeltaNet（軽い計算）とTransformer（重い計算）を組み合わせ、さらにMoE（混合エキスパート）により必要な回路だけを動かすことで、超長文脈（最大1Mトークン）を高速処理する。

---

## 4. ファインチューニング（追加学習）の違い

モデルの構造が進化するにつれて、画像を含むファインチューニングの難易度やアプローチも変化しています。

| 項目 | 従来モデル（LLaVA, Qwen 2-VL等） | 最新ネイティブモデル（Qwen 3.5等） |
| :--- | :--- | :--- |
| **主な学習対象** | プロジェクター ＋ LLM本体 (Visionは凍結) | LLM本体 ＋ 統合層 ＋ Vision層 (Vision-LoRA) |
| **画像のリサイズ** | 固定サイズ（リサイズやクロップで歪む） | **動的解像度**（元の比率と鮮明さを保持） |
| **位置情報の処理** | 単純な1D/2D位置エンコーディング | **MRoPE**（空間と時間の複雑な位置計算） |
| **推奨ツール** | 汎用的な学習スクリプトでも対応可能 | **SWIFT** や **LLaMA-Factory** など専用対応が必須 |

### Qwen 3.5 における学習のポイント
画像とテキストが深く統合されているため、LLM側だけを学習させると画像認識力が落ちる「破滅的忘却」が起きやすくなります。そのため、Vision Encoder側にもLoRAを適用する **Vision-LoRA** や、特定のMoEルーターを調整するアプローチが推奨されます。
