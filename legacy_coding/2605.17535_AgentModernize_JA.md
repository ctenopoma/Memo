# AgentModernize: マルチエージェントLLMと振る舞い仕様グラフによるレガシーシステムの事業ロジック保存

**著者:** Sheikh Nazib Ahmed, Marnim Galib (University of Texas at Arlington)  
**arXiv:2605.17535v1 [cs.SE] 2026年5月17日**

---

## 要約(Abstract)

レガシーシステムの近代化は事業ロジックを破壊する。時々ではなく、しばしば。ほとんどのツールとLLMベースのアプローチは近代化を構文変換として扱う:COBOLをJavaに変換し、PL/SQLをPythonに置き換え、出荷する。しかし、本番システムを稼働させ続ける暗黙のルール、エッジケース処理、モジュール間の制約が失われる。誰も本番環境で何かが失敗するまで気づかない。

本研究では**AgentModernize**を提案する。4つの専門エージェント(抽出・仕様化・コード生成・検証)が協働し、近代化を構文変換ではなく**振る舞い保存問題**として扱う。中核となる中間表現「**Behavioral Specification Graph (BSG)**」は、抽出されたビジネスロジックをコード生成前に明示的かつ検査可能にし、LLMが「理解した内容」と「生成する内容」の間に信頼境界を形成する。

**LegacyModernize-8**ベンチマーク(通信・銀行ドメイン8シナリオ)で評価した結果:
- フィードバック付きAgentModernize(Full)のみ、全モデルで非ゼロの振る舞い等価率(BER)を達成
- SP-LLM・CoT-LLMは全シナリオで**0.0%**
- AgentModernizeのBER: GPT-4o-mini 9.4%、GPT-4o 8.1%、GPT-5.3-codex 19.4%
- BSGは金標準ルールの**91.2%**を捕捉(=ボトルネックは抽出ではなくコード生成)
- 規制産業向けに、ビジネスルール台帳・BSG・等価性レポートという追跡可能な成果物を提供

## I. はじめに(Introduction)

通信事業者、金融機関、医療提供者は、20〜30年前に構築されたシステムで中核事業を運営している。COBOLバッチジョブが請求を処理し、PL/SQL手続きが契約ロジックを強制し、シェルスクリプトがプロビジョニングワークフローを指揮している。これらのシステムは機能している――そしてそこに組み込まれたビジネスルールは、本番経験、バグ修正、規制対応を通じて何年にも渡って洗練されている。しかし維持は高コスト、現代プラットフォームとの統合は困難、そしてシステムを書いた技術者の退職により脆弱化が進む。

近代化は自明の対応に見える。しかし見落とされやすい問題は、ほとんどの近代化作業が、本来保存すべき最も重要なもの――**振る舞い意味論**――を破壊することである。レガシーシステムのビジネスルールはほとんど文書化されていない。それらは制御フロー、条件分岐、例外ハンドラ、設定ファイルの中に埋もれている。行単位のCOBOL→Java翻訳はコンパイルして実行できても、エッジケース処理を静かに破壊し、検証ロジックを変更し、決して文書化されなかった制約を欠落させる可能性がある。

私たちは実例を見ている: 通信プロビジョニングワークフローで、構文的に正しい翻訳がコンパイルを通過したにもかかわらず、15年間本番稼働していた「停止口座の免除」処理を静かに削除してしまった。結果は現代的に見えるが動作が異なるシステム。

**Large Language Models (LLM)** はこの問題に対する自然なツールを提供する――レガシーコードを解析し、意図を推論し、現代的な等価物を生成できる。しかし単に「このレガシーコードを現代APIに変換してください」とLLMにプロンプトするのは脆弱である。理由は少なくとも3つ:
1. コンテキストウィンドウではレガシーコードベース全体を一度に保持できない
2. 単一の生成パスでは、振る舞いが保存されたことを検証する機構がない
3. 単一の大型プロンプトは「理解・仕様化・変換・検証」を一体化させ、分離のメリットが失われる

### 提案手法: AgentModernize

AgentModernizeはレガシー近代化を4つのエージェント処理フェーズに分解する:

1. **Legacy Analyzer**: レガシー成果物から暗黙のビジネスルール、制御フロー、運用制約を抽出
2. **Specification Generator**: 抽出したロジックを「事前条件・事後条件・不変条件」を持つ構造化中間表現「Behavioral Specification Graph (BSG)」に符号化
3. **Modernization Transformer**: BSGから現代的なサービス実装を生成、振る舞い契約を保存
4. **Equivalence Validator**: 生成されたコードをBSGに対して検証(自動テスト生成・差分トレース解析)、失敗情報をTransformerに繰り返し修正指示

**核心的洞察**: BSGは抽出と生成の間の「ガラス箱(glass box)」として機能する。コード生成前に、抽出されたビジネスロジックを明示的かつ検査可能にする。バリデーターが振る舞い乖離を検出すると、全再生成ではなく**的を絞った修正ループ**を起動。フィードバックループはパイプライン全体で最も重要な機構であることが判明した(§V, Table VI)。

## II. 関連研究(Related Work)

### A. レガシーシステム近代化

Comella-Dorda ら (2000) は「ラッピング」から「全面再構築」に至る近代化戦略を分類した。モデル駆動アプローチ(MoDisco, KDM/ADM標準)はレガシー知識を形式化しようとしたが、構築作業が近代化自体のコストに匹敵し、実用的ではない。

産業ツール(IBM Rational Asset Analyzer、Micro Focus Enterprise Analyzer、COBOL→Javaトランスパイラ)は構文変換には長けているが、**「コンパイル・実行できる」は「振る舞い等価」ではない**。これらは翻訳システムが元のシステムと同じく動作することを検証せず、振る舞い等価性指標も公表していない。本研究はこの空白を対象とする。

### B. LLMとソフトウェア工学

LLMはコード生成、バグ検出、コード審査、プログラム修復で成功を示している。Pan ら (2024) の最も関連ある研究は「GPT-4は構文的に正しいクロス言語翻訳を生成するが、複雑な制御フローで意味バグを導入する」と指摘。これは私たちの実験観察と完全に一致: 全ての単発プロンプトベースラインはコンパイル・実行できたが、振る舞いテストでは**0.0%**。

共通パターン: 全ての先行研究は近代化を**単一パスタスク**として扱う。1つのモデル、1つのプロンプト、1つの出力。中間表現なし、検証なし、フィードバックなし。自己完結ユーティリティなら機能するが、13の相互作用ビジネスルール例外ハンドラに埋もれた制約を持つ280行のCOBOL手続きにはそうではない。

### C. マルチエージェントLLMシステム

MetaGPT、ChatDev、AgentCoder、Self-Refineなどは全てグリーンフィールド開発を対象とする。新規コード作成から既存コードの暗黙ロジック抽出・検証を経由した近代化まで、根本的に異なる問題である。

### D. 振る舞い等価性

形式的プログラム等価性(二分化、トレース等価、観察的等価)は検証アプローチの理論的根拠を与える。全形式検証は実世界システムではしばしば困難で、実務家は軽量代替案を用いる:
- **差分テスト**: システムバージョン間で出力を比較
- **メタモルフィックテスト**: 既知の入力変換が期待する出力変換を生成するか検査

**本研究の差別化点**: 等価性バリデーターは**レガシーコードから直接ではなく、BSG仕様からテストオラクルを生成**する。テスト対象(翻訳)とテスト方法(そのコード自身)の循環を回避する。これはほとんどの自動テスト手法が見落とす微妙だが重要な点。

## III. AgentModernizeフレームワーク

### A. 概要

フレームワークはレガシー成果物バンドル(ソースコード、設定、DB スキーマ、文書)を入力とし、近代化サービス実装と振る舞い保存忠実性レポートを出力する。4つのエージェントが順序良く処理、共有パイプラインステートから読み書き。

**重要な設計選択**: パイプラインは純粋フィードフォワードではない。Equivalence Validator (Agent 4) が振る舞い乖離を検出すると、Modernization Transformer (Agent 3) に**的を絞った修正指示**をフィードバックする。このループは最大3回実行可能で、人間介入なく自己修正可能。

### B. レガシー成果物バンドル

入力 L = {S, C, D, N}:
- **S**: ソースコードファイル(COBOL、PL/SQL、シェルスクリプト)
- **C**: 設定ファイル(JCL、パラメータファイル、cronスケジュール)
- **D**: データベーススキーマ(DDL、copybook、データ辞書)
- **N**: 自然言語文書(オプション、runbookやコメント)

### C. Agent 1: Legacy Analyzer

**Phase 1a - 構造解析**: プログラムエントリポイント、モジュール境界、データフロー、外部依存を特定。手続き型レガシーではPERFORM/CALL階層、ファイルI/O操作、DB相互作用を同定。

**Phase 1b - ビジネスルール抽出**: 条件ロジック、検証ルール、計算公式、状態遷移、例外処理パターン。各抽出ルールに注釈:自然言語説明、ソース位置、入出力変数、信頼度スコア。

**Phase 1c - 制約発見**: データ型制約、値範囲、参照整合性、時系列順序、ビジネス固有不変量(例:「注文合計 = 行アイテム合計」)。

**出力**: Business Rule Inventory (BRI) ――抽出されたすべてのルール、制約、メタデータを含むJSON。

### D. Agent 2: Specification Generator

BRIを**Behavioral Specification Graph (BSG)**に変換――AgentModernizeの核となる中間表現。

**定義 1 (BSG)**: 有向非環グラフ G = (V, E, Pre, Post, Inv)
- **V**: 操作ノード集合(例「注文検証」「税計算」)
- **E ⊆ V × V × L**: ラベル付きエッジ(L = {sequence, conditional, parallel, error})
- **Pre: V → P**: 各ノードの事前条件(論理述語)
- **Post: V → P**: 各ノードの事後条件
- **Inv: G → P**: 全体不変条件

各ノードはさらに以下で注釈:
- inputs(v): 必要な入力データフィールド・型
- outputs(v): 生成される出力フィールド・型
- rules(v): BRIのビジネスルールID集合
- error_behavior(v): エラー時の期待動作

**具体例(S1, Order Validation)**:
```json
{
  "operation": "ValidateDisconnectOrder",
  "source_location": "ORDER_VALIDATION.cob:118-142",
  "preconditions": [
    "account_status in ['ACTIVE', 'SUSPENDED']",
    "order_type == 'DISCONNECT'"
  ],
  "postconditions": [
    "suspended accounts proceed for disconnect orders",
    "non-disconnect orders rejected for suspended"
  ],
  "invariants": ["order_total == sum(line_items)"],
  "confidence": "high"
}
```

このノードは暗黙ルールを捕捉――停止口座は通常ブロックされるが、切断注文は免除される。BSGがこれを明示的にし、コード生成前に検査可能にする。

### E. Agent 3: Modernization Transformer

BSGから現代的なサービス実装を生成。以下の変換原則で動作:

1. **Operation-to-Endpoint Mapping**: 各BSG操作ノード → 現代システムの離散関数/APIエンドポイント
2. **Contract Preservation**: BSG事前/事後条件 → 入力検証、出力アサーション、エラーハンドリング
3. **Data Model Derivation**: BSGの入出力注釈 → 現代型付きデータモデル(Pydantic等)
4. **Flow Preservation**: BSGエッジラベル → 現代実装のオーケストレーション

**出力**: Modernized Service Package
- APIエンドポイント実装(Python/FastAPI)
- 検証ルール付きデータモデル
- BSG制御フロー保存のオーケストレーションロジック
- BSGエラー動作に合致するエラーハンドリング
- 各エンドポイントをBSGノードに対応付けたドキュメント

### F. Agent 4: Equivalence Validator

現代実装がBSG指定の振る舞い等価性を保存していることを検証。3タイプ:

**Type 1 - Contract Verification**: 各BSGノードについて、対応するエンドポイントへのテストケース生成。事前条件が強制され、事後条件が成立、不変条件がワークフロー全体で保持されることを検証。

**Type 2 - Boundary Testing**: BSG制約に基づく境界テスト(最小/最大値、空入力、null処理、型境界)。

**Type 3 - Differential Trace Analysis**: BSGからエンドツーエンドワークフロートレースを構築し、現代実装に対して実行。実出力とBSG指定出力を比較。

**出力**: Behavioral Equivalence Report (BER)
- ノード単位の等価ステータス(PASS/FAIL/PARTIAL)
- 全体振る舞い等価率
- 失敗テストと根本原因分析
- 修復推奨

**フィードバックループ**: BER が振る舞い乖離を特定すると、パイプラインは対象修正指示(失敗テストと期待動作)をTransformerに再呼び出し。最大イテレーション数はデフォルト3。

## IV. 評価設計

### A. 研究質問

- **RQ1 (振る舞い等価性)**: AgentModernizeはレガシーシステムの振る舞い意味論をどの程度保存するか?
- **RQ2 (ビジネスルール保存)**: 個別ビジネスルールをどの程度抽出・保存するか?
- **RQ3 (努力削減)**: 単発LLMベースラインと比べ手作業削減は?
- **RQ4 (エージェント貢献度)**: 各エージェントの個別貢献は?
- **RQ5 (BSG品質)**: ビジネスルール捕捉精度は?
- **RQ6 (モデル感度)**: LLMモデルサイズで品質は変わるか?

### B. ベンチマーク: LegacyModernize-8

通信ドメイン7シナリオ+クロスドメイン銀行シナリオ。

| ID | シナリオ | ドメイン | 複雑度 | 規則 | 行数 |
|----|---------|---------|--------|------|------|
| S1 | Order Validation | 注文 | 中 | 12 | 248/195 |
| S2 | Billing Disputes | 請求 | 高 | 14 | 245/280 |
| S3 | Service Activation | プロビジョニング | 高 | 13 | 230/210 |
| S4 | Circuit Inventory | インベントリ | 高 | 15 | 280/310 |
| S5 | Fault Escalation | 障害管理 | 高 | 12 | - |
| S6 | Contract Renewal | 契約 | 高 | 10 | - |
| S7 | Account Migration | アカウント | 超高 | 13 | - |
| S8 | Bank Transactions | 銀行(COBOL) | 超高 | 15 | - |

各シナリオ: 明示ルール+3つ以上の暗黙ルール。複雑度はドメイン専門家が評価した条件分岐・外部依存・状態遷移の数。

### C. 指標

**M1 - Behavioral Equivalence Rate (BER)**:
```
BER = (合格テスト数 / 合計テスト数) × 100%
```

**M2 - Business Rule Preservation Score (BRPS)**:
```
BRPS = (保存されたルール数 / 総ルール数) × 100%
```
ルールは(1) BSGに出現 AND (2) 現代コードで強制 の両条件で「保存」と判定。

### D. ベースライン

- **B1 - SP-LLM (Single-Prompt)**: レガシー成果物全体+現代化指示を単一プロンプト。中間表現なし、検証なし。
- **B2 - CoT-LLM (Chain-of-Thought)**: 同じく単一プロンプトだが「まず分析、次にAPI設計、次に実装」の思考連鎖付き。
- **B3 - AgentModernize (No Feedback)**: 完全パイプラインだがフィードバックループ無効化。反復精緻化の貢献を測定。

### E. 実装詳細

- **LLM**: GPT-4o-mini (デフォルト)、GPT-4o (比較用)、GPT-5.3-codex (最先端)
- **オーケストレーション**: LangGraph (Python)
- **テスト実行**: pytest
- **レガシー言語**: COBOL/PL/SQL
- **近代化ターゲット**: Python (FastAPI)
- **温度**: 0.2 (抽出・仕様)、0.0 (コード生成・評価)
- **フィードバック最大反復**: 3回
- **評価プロトコル**: 各シナリオ・各手法で3試行、温度0.0で決定的

### C. RQ4: アブレーション研究(Table VI)

| 構成 | SP-LLM | CoT-LLM | AM (No FB) | Full AM |
|-----|--------|---------|------------|---------|
| 平均BER | 0.0 | 0.0 | 0.0 | **9.4** |
| 非ゼロシナリオ | 0/8 | 0/8 | 0/8 | **3/8** |

**重要な発見**: **フィードバックループは必須であり、オプションではない**。無しではAgentModernizeは全ベースラインと同じ0.0%。マルチエージェント分解(抽出→BSG→生成)だけではコンパイル・実行は可能だが振る舞いテスト合格はない。Validatorが失敗情報をTransformerにフィードバックして初めて期待動作と合致。

しかしフィードバックは必要だが十分ではない。3/8シナリオでのみ非ゼロ。S2~S4, S6, S7では生成コードのAPI構造が金標準期待値と乖離しすぎ、反復パッチ適用では埋められない。**フィードバックループは点エラー(誤った定数、欠落条件)は修正できるが、構造ミスマッチ(異なるAPI形状、誤ったエンドポイント意味論)は克服できない**。

**重要**: 再現率91.2%, 適合率89.7%。大多数の金標準ルールを捕捉、にもかかわらず端末BER 9.4%。**72~82ポイントの乖離は、ボトルネックが抽出ではなく***Modernization Transformer**(Agent 3)***であることを示す**。

Transformer の code generation と structural alignment が weak link。バリデーターが「エンドポイント形状が金標準期待値と大きく乖離」と報告すると、いかに正しく実装されたビジネスロジックでも標準テストは合格しない。

| S8 | 33.3 | 0.0 | 35.6 |
| **平均** | **9.4** | **8.1** | **19.4** |

**結論**:
- 強いモデルはより高いBER: codex 19.4% > 4o 8.1% > mini 9.4%
- S1は全モデルでゼロ超(最良25%~58.3%)
- 各モデルは異なるシナリオで成功(相補的強度)
- **どのモデルでも、SP-LLM/CoT-LLMは全8シナリオで0.0%**. パイプラインとフィードバックが極めて重要

---

## VI. 議論

### A. BSGは信頼境界

BSGは最も熟慮された設計選択。上流(抽出、信頼度スコア付き)と下流(生成、仕様契約で検証可能)の境界。

RQ5結果はこれを支持:
> BSGが金標準ルール91.2%を捕捉しながら、端末BER 9.4% (19.4% with codex)。72~82ポイント乖離は、ボトルネックが抽出でなくコード生成であることを示す。

**実務家への示唆**: Agent 2 (Specification Generator) 後にBSGを検査してルール確認できれば、問題は下流のコード生成にあると判断可能。

### B. フィードバックループが決定的

全3モデルで、Full AgentModernize with feedback のみが非ゼロ平均BER を達成 (9.4%, 8.1%, 19.4%)。SP-LLM・CoT-LLM は全backbone で 0.0%。フィードバック無し AgentModernize は mini/codex で 0.0%, 4o でも 5.6% 平均 (S1 44.4% のみ)。

**パイプライン効果**: 初期コードが80%正しい (良いモデル) なら、フィードバックループが残り20%の一部を埋める。初期が40% (弱いモデル) なら、ループは大きすぎる乖離に対処できない。**アーキテクチャと モデルが協働する ―― どちらか一方では不十分**。

### C. 展望と限界

**What this approach cannot do yet**:
- 評価シナリオ: 100~310 LOC。実レガシーは桁違いに大きい。
- 対応には: Legacy Analyzer chunking, hierarchical BSGs, incremental validation が必要 (工学課題、根本的限界ではない)。

**Practical takeaway**: 
- 予算モデル (mini): パイプラインは唯一の非ゼロ BER 経路
- 最先端 (codex): 最高 BER (19.4%)
- **規制ドメイン** (通信・銀行): BSG, rule inventory, equivalence report は single-prompt 手法が提供できない監査証跡

---

## VII. 脅威妥当性

- **内的妥当性**: 8シナリオは合成だが(S8は異ドメイン)、現実的複雑性を反映。金標準は単一専門家作成(複数注釈者で強化が望まれる)
- **外的妥当性**: 通信中心(銀行1シナリオ)。医療・政府等の他ドメイン要確認。COBOL/PL/SQL評価; Python 他言語への汎化未確認。
- **構成妥当性**: テストベース検証は辺境エッジケースを逃す可能性。形式検証はスケール困難。
- **信頼性**: LLM 非決定性を温度 (0.0/0.2) で抑制。3試行。3モデルで検証。
- **LLMジャッジ偏り**: Validator 自体が LLM。金標準テストは独立生成・pytest で決定的実行なので、BER は評価プロトコル依存。

---

## VIII. 結論

**レガシー近代化は翻訳問題ではなく、振る舞い保存問題である。**

AgentModernize はこれを実装し、結果は明確: フィードバックループがすべてを変える。
- SP-LLM, CoT-LLM: 全8シナリオ全backbone で 0.0% BER
- AM (No Feedback): mini/codex で 0.0%, 4o でも 5.6% 平均
- **Full AM with Feedback**: 9.4% (mini), 8.1% (4o), 19.4% (codex) ― 全backdrop で唯一の非ゼロ

BSGは91.2%のルール捕捉し、ボトルネックがコード生成 (抽出ではない) と指摘。

**3つの方向が有望**:
1. Hierarchical BSGs for > 300 LOC codebases
2. Ensemble strategies selecting best model per scenario (3モデル結果から 4/8 は可能)
3. **Human-in-the-loop study**: ドメイン専門家が BSG 検査で modernized code をより信頼するか確認 ―― 最重要問題
