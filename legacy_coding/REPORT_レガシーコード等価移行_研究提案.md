## 1. 事業的課題（論文に記載された事実）

- **保守人材の枯渇とコスト上昇。** レガシー言語(COBOL, Fortran, PL/SQL, Assembly)に精通した開発者は減少し、習得コストが上昇し続けている。米連邦政府のレガシーは稼働30〜60年に及ぶ(PL/SQL→Java)。AgentModernize も「コードを書いた技術者が退職し、システムが脆弱化している」と指摘。
- **手作業移行は超高コスト。** Commonwealth Bank of Australia は COBOL→Java 移行に約5年・約7.5億ドルを投じた(PL/SQL→Java が引用)。LegacyTranslate / PL/SQL→Java の対象は単一金融機関で**約250万行**の PL/SQL。
- **ハードウェア進化への追随。** 科学計算 Fortran は CPU 同質環境向けに最適化されてきたが、GPU 等ヘテロ環境はネイティブ Fortran バインディングを欠き、移植が急務(Fortran→Kokkos)。
- **安全クリティカル領域での慎重姿勢。** 開発者は安全クリティカルなレガシーで「意図しないバグが重大リスクになる」ため AI 翻訳に慎重(PL/SQL→Java)。C→Rust は安全性ゆえに産業利用の動機が強いが、産業実用には対話用途より高い信頼性が必要(C-to-Rust, Bosch)。
- **規制産業での監査要件。** 通信・金融・医療のコア業務が20〜30年前のシステムで稼働。規制産業では「ビジネスルール台帳・仕様・等価性レポート」という追跡可能な成果物が求められ、単発プロンプト方式では提供できない(AgentModernize)。

> 研究提案への含意: 「人材枯渇 × 高コスト × 安全/規制要件」が市場ニーズの三本柱。制御ロジック(規制・安全)と数値計算(HPC・科学)はいずれも**「動けばよい」では済まず、等価性の証拠と監査証跡が事業価値**になる領域。

## 2. 技術的課題（論文に記載された事実）

### 2.1 直接（ワンショット）翻訳は信頼できない
- LLM 翻訳の正答率は **2.1〜47.3%** にとどまる(Pan et al. を Fortran→C++ / Intent-Preserving が引用)。翻訳プログラムの**最大77.8%がコンパイル不能か誤結果**(Intent-Preserving が引用)。
- 高レベルのアルゴリズム構造は保たれても、**型処理・I/O 意味論・境界条件・ループ境界・数値規則などの低レベル詳細を取りこぼす**。数値パースや除算規則の差で off-by-one や境界チェックが壊れる(Intent-Preserving)。

### 2.2 「コンパイル・実行できる ≠ 振る舞いが等価」
- AgentModernize の中心主張:行単位の COBOL→Java はコンパイル・実行できても、エッジケース処理・検証ロジック・制約を**サイレントに破壊**する。実例として、通信プロビジョニングで**15年間本番稼働していた「停止口座の免除」処理が、構文上正しい翻訳で静かに欠落**した。
- Pan et al.: GPT-4 は構文的に正しい翻訳を作るが、**複雑な制御フローで意味バグを混入**(AgentModernize が引用)。単発プロンプトの各ベースラインは「コンパイル・実行は通るが振る舞いテストは 0.0%」。
- 既存の産業ツール(IBM Rational Asset Analyzer, Micro Focus Enterprise Analyzer, COBOL→Java トランスパイラ)は構文変換は得意だが、**振る舞い等価性を検証せず、等価性指標も公表していない**(AgentModernize)。

### 2.3 ビジネスルール/意図が暗黙知である
- ビジネスルールはほとんど文書化されず、**制御フロー・条件分岐・例外ハンドラ・設定ファイルの中に埋もれている**(AgentModernize)。
- 大規模レガシーは**一貫した文書も自動テストも欠く**(PL/SQL→Java の VT システム=250万行、AgentModernize)。

### 2.4 本番フレームワーク・API への適合
- 直接 LLM 翻訳は構文的に妥当でも**既存の本番フレームワーク・共有 API に統合できず実用に至らない**。API グラウンディングが無いと**生成コードは本番フレームワークでほぼコンパイルしない**(LegacyTranslate)。

### 2.5 等価性「検証」自体が難しい
- 検証はテストスイートに依存するが、**ユニットテストは不完全で重要入力を欠き、誤った等価判定(偽の等価)**を生む。ファジングは無効入力を生み**偽の非等価**を生む(MatchFixAgent)。
- 検証ツールは**1言語ペア専用の実装に閉じがち**(ALPHATRANS 10,859 行、OXIDIZER 19,052 行)で、言語ペア数が二乗で増えるためスケールしない(MatchFixAgent)。

### 2.6 評価パイプライン由来の「偽の失敗」
- 報告される失敗の相当数は**翻訳の誤りではなく、コンパイルフラグ欠落・ライブラリ未リンク・ランタイム未設定など評価環境の不備に由来**する。5言語・3ベンチ・**6,164翻訳**の分析で、意味的に正しいコードが誤って「不正」と分類される偽陰性を多数特定(Beyond Translation Accuracy)。

### 2.7 数値計算に固有の難しさ（本研究の核心）
- 科学計算出力はテキストと数値が混在し、**文字列完全一致では本来同じ出力を「大きく異なる」と誤判定**する(例: `0.00000000` vs `0.0`、`832040` vs `832040.0`)。Fortran→C++ はこれに対し、出力をトークン化→float 変換→**小数4桁丸め**→Jaro-Winkler 類似度で比較する許容差付き評価を導入。
- すなわち**「ビット完全一致」を等価の基準にできない**。ここに、ユーザの問題意識(レガシー独自ロジックが数値誤差を生む可能性 → レガシー出力を絶対的正解(オラクル)にできない)と直結する論点がある。

> **重要なギャップ(研究提案の起点)― 2026-06-16 追加調査により訂正:** フォルダ内の数値系論文(Fortran→C++, Fortran→Kokkos)はいずれも**レガシー出力を許容差付きで「正解」とみなして**比較している。
>
> ただし「物理式など普遍的知見をオラクルとする」アイデア自体は、**コード翻訳の外側に確立した先行分野が存在する**(当初「見当たらない」と書いたのは調査不足による誤りで、以下に訂正):
> - **メタモルフィックテスト(MT)/メタモルフィック関係(MR):** 1998年以降「オラクル問題」を解くために発展。正解データ無しに、プログラムが満たすべき性質(対称性・スケール則・保存則等の物理/数学的不変量を含む)で出力を検証。1998–2015年のMT応用の1/4超が科学計算ソフト(PDE・モンテカルロ等)。
> - **計算物理のコード検証(Code Verification):** Method of Manufactured Solutions (MMS)、簡略化物理の解析解照合、収束次数解析。物理/数学知見を直接オラクル化する標準手法(Roache 他)。
> - **MTのLLM応用:** コード生成LLMの安定性検証(対称MR)、LLMプログラム修復などに2024–2025年に適用例あり。
>
> **したがって真のギャップは「不在」ではなく「未統合」である:** これら普遍知見オラクル(MT/MR・MMS・コード検証)と、LLMによるレガシーコード翻訳/マイグレーションが**別々の文献に分かれており、両者を統合した研究 ―― 特にレガシー数値コード→Python 移行の等価性オラクルに MT/MMS を据える研究 ―― は本調査範囲では見当たらない**(「Fortran→Python × メタモルフィック関係 × 浮動小数点等価」での検索でも該当論文なし)。フォルダ内マイグレーション論文はどれも等価性を「レガシー出力±許容差」か「テスト」で定義し、MT/MMS を採用していない。これは**橋渡し型(bridge)の研究機会**であり、先行分野が揃っている分、説得力・実現性が高い。
>
> 参考: Metamorphic testing (Wikipedia / Exploratory MT for Scientific Software, PMC7252536); MMS (Springer 2017, 978-3-319-70766-2_12; NAFEMS code-vs-solution verification); 対称MRによるcode-gen LLM検証 (J. Syst. Softw. S0164121224003741); MT×LLM program repair (arXiv 2410.07516)。

## 3. 打ち手（論文が採用・検証した技術）

論文群を貫く**収束パターン**が3つある。

### 打ち手A: 「生成前の中間仕様」を立てる（contract-first / glass-box）
直接翻訳ではなく、**言語非依存の中間表現を先に確定**してから目的言語コードを生成する。
- **AgentModernize / Behavioral Specification Graph (BSG):** 事前条件・事後条件・不変条件を持つ構造化中間表現。「LLM が理解した内容」と「LLM が生成する内容」の間に**人間が点検できる信頼境界(glass box)**を置く。**コード生成前に**抽出済みビジネスロジックを明示・検査可能にする。
- **Intent-Preserving Pipeline:** I/O 契約・データ構造・**数値規則**・ループ境界・出力形式を捉えた**言語中立アルゴリズム**を推論時に先に固定し、その不変条件を満たす慣用コードを生成。LLM を「トークン生成」から「契約駆動の実体化(contract-driven realization)」へ転換。
- **BabelCoder:** ソースから推論した NL 仕様とテストで翻訳を精緻化。**PL/SQL→Java** は「ドメインモデル+少数例」、**サンプルは量より類似度が効く**ことを報告。

### 打ち手B: 「生成&検査 + フィードバックループ」で自己修復する
生成物をコンパイル/実行/差分で自動検査し、失敗時に LLM へ再プロンプトして反復修復。
- **C-to-Rust:** generate-and-check パターン。コンパイル可否は Rust コンパイラのエラーを、振る舞い等価性は**差分ファジング**で検査し、否定結果で再プロンプト。**フィードバックループを使うとモデル間の性能差が縮小**し、堅牢性も上がる(検証可能性が「保証付き LLM 工学」を可能にする)。
- **Fortran→Kokkos:** 翻訳・検証・コンパイル・実行・テスト・デバッグ・最適化を**専門エージェントが分担**する完全自動ワークフロー。
- **AgentModernize:** 4エージェント(抽出/仕様/生成/検証)。検証器が振る舞い乖離を検出すると**全再生成ではなく的を絞った修正ループ**を起動。「**フィードバックループがパイプライン全体で最も重要な機構**」と結論。
- **LegacyTranslate:** 初期翻訳/API グラウンディング/精緻化の3エージェント。Refinement はコンパイラフィードバックと API 提案で反復改善。

### 打ち手C: 等価性検証・修復を独立タスクとして強化
- **MatchFixAgent:** 言語非依存・LLM ベースの等価性検証&修復。等価性判定を制御フロー・データフロー等の**サブタスクに分割**して一貫した意味解析を行う、マルチエージェント構成。
- **API グラウンディング(LegacyTranslate):** API 知識ベースから関連エントリを検索し、本番 API に整合させる。
- **評価の健全化(Beyond Translation Accuracy):** 構成情報(フラグ・リンカ・タイムアウト)を意識した**透明で再現可能な評価標準**の必要性。
- **人間の役割設計(Human-AI Partnership):** 人間を**「ディレクター」かつ「レビュア」**と位置づけ、モデルの結果類型に応じた**信頼フレームワーク**を提示。

## 4. 評価（論文が用いた指標と定量結果）

### 4.1 評価指標の系譜
| 指標 | 内容 | 出典 |
|---|---|---|
| Compilation accuracy | コンパイル成功率 | Fortran→C++, LegacyTranslate |
| CodeBLEU | 人手翻訳との n-gram/AST/データフロー類似 | Fortran→C++ |
| Output similarity（許容差付き） | 出力をfloat化・4桁丸め後 Jaro-Winkler | Fortran→C++ |
| Test pass / pass@1 | テスト通過率 | LegacyTranslate, MigrationBench, Intent-Preserving |
| Approximate Functional Equivalence | 近似的機能等価 | MigrationBench |
| Behavioral Equivalence Rate (BER) | 振る舞い等価率 | AgentModernize |
| Business rule preservation（明示/暗黙別） | ルール保存率 | AgentModernize |
| (In)equivalence verdict 正確性 / 修復率 | 等価判定と修復成功 | MatchFixAgent |

### 4.2 主要な定量結果（事実）
- **MigrationBench:** Claude-4.5-Sonnet + 自前エージェントで selected サブセットの最小移行 **71.67%**、最大移行 **53.33%**(pass@1)。データセットは全5,102 / 102 / 300 リポジトリ。「近似的機能等価」をキーワードに掲げる。
- **LegacyTranslate:** 初期翻訳エージェント単体で**コンパイル可 45.6% / テスト通過 30.9%**。API グラウンディング+精緻化で**コンパイル +8% / テスト通過 +3%**。教訓:API グラウンディング無しでは本番でほぼコンパイルせず、検索品質が機能正しさを左右し、モデル能力がフレームワーク制約処理を決める。
- **AgentModernize:** LegacyModernize-8(通信・銀行8シナリオ)で GPT-4o-mini/GPT-4o/GPT-5.3-codex を温度0.0・3試行で評価。**フィードバック付き Full 構成だけが全バックボーンで非ゼロ BER**。単発(SP-LLM)・CoT は全シナリオ・全モデルで **0.0%**。Full の平均 BER は 9.4%(mini)/8.1%(4o)/19.4%(codex)、最大75.0%。**BSG はゴールドルールの 91.2% を捕捉**(=ボトルネックは抽出でなくコード生成)。追跡可能な成果物が監査証跡になる。
- **MatchFixAgent:** 翻訳ペアの **99.2%** に等価/非等価判定を出力。先行研究と判定が割れた場合、**60.7% は本手法が正しい**。非等価の**修復成功 50.6%(先行 18.5%)**。
- **Intent-Preserving Pipeline:** Avatar/CodeNet で Python↔Java、5 LLM。アルゴリズム中間表現でマイクロ平均正答率 **67.7%→78.5%(+10.8pt)**。字句/トークンエラーを100%除去、ランタイム依存・エントリポイント失敗を 78.4% 低減。一方、型/オーバーロード解決などは残存。
- **BabelCoder:** 4ベンチで平均正答率 **94.16%**、94% のケースでベースライン比 +0.5〜13.5%。
- **C-to-Rust(Bosch):** フィードバックループ無しでは**モデル選択の影響が大**だが、ループ有りで**モデル差が縮小**。振る舞い保存的コード摂動(perturbation)の多様性が性能を上げることも。
- **Fortran→Kokkos:** GPT-5 / o4-mini-high は全カーネル・全ハードウェア区画でフルパイプライン完走し、**Fortran ベースラインを超える最適化コード**を**数ドル**で生成。open-source の Llama4-Maverick はしばしば機能コードを作れず。**有償↔OSS の性能差**を明示。
- **Fortran→C++:** **モデルのパラメータ数増加にほぼ比例してコンパイル精度が向上**。CodeBLEU だけでは LLM 間差が見えにくく、出力類似度等の併用が必要。

### 4.3 評価上の落とし穴
- **偽の失敗(Beyond Translation Accuracy):** 6,164翻訳の分析で、コンパイルフラグ欠落・誤リンカ・非現実的タイムアウトにより**意味的に正しい翻訳が「失敗」と誤計上**。パイプライン由来の失敗(全モデル共通)とモデル依存の失敗を区別すべき。

## 5. 想定される展望（論文の主張に基づく）

- **大規模レガシー近代化の自律化が現実味。** Fortran→Kokkos は「かつて専門家が数週間・高コストで行った作業を、有償 LLM で数時間・数ドル・人手介在なしに」実現できると主張し、科学発見の加速・重要レガシー延命・次世代スパコンへのアクセス拡大を展望。
- **エージェント分業+検証ループが標準アーキテクチャに。** 複数論文(AgentModernize/Fortran→Kokkos/BabelCoder/LegacyTranslate/MatchFixAgent)が独立に「翻訳・テスト・検証・修復」の役割分担へ収束。サーベイ(Code Agents)も test-and-repair パラダイムを整理。
- **中間仕様が「信頼境界」「監査証跡」として制度化。** BSG/言語中立仕様は規制産業の監査要件に応える成果物になりうる(AgentModernize)。人間はディレクター/レビュアとして残る(Human-AI Partnership)。
- **検証の汎用化。** 言語ペア専用実装の限界を、LLM ベースの言語非依存検証(MatchFixAgent)が突破しうる。
- **残る限界(誇張への警告):** BER の絶対値は依然 modest(平均一桁台〜)で、ボトルネックは抽出でなく**コード生成側**(AgentModernize)。型解決等は中間表現でも残る(Intent-Preserving)。OSS モデルは産業要件に未達のことが多い(Fortran→Kokkos)。評価標準も未成熟(Beyond Translation Accuracy)。

### 6.1 ユーザの2本柱への対応づけ
- **制御ロジック(=ビジネスルール保存):** ①②④⑤を厚くカバーするのは **AgentModernize(BSG+検証ループ+監査証跡)**、**LegacyTranslate(API グラウンディング)**、**MatchFixAgent(言語非依存検証)**。「要件を作り直し人が承認しながら」というユーザ方針は、**AgentModernize の glass-box 中間表現** と **Human-AI Partnership の director/reviewer モデル**に理論的裏付けがある。
- **数値計算(=数値等価性):** ③④をカバーするのは **Fortran→C++(許容差付き出力比較)**、**Fortran→Kokkos(実行ベース検証+性能移植)**、**C-to-Rust(差分ファジング)**、**MigrationBench(近似機能等価)**。ただし**全てレガシー出力を許容差付きの「正解」とみなす**前提。

### 6.2 本調査で確認された空白(研究提案の差別化ポイント)
1. **物理式・普遍的知見をオラクルとする数値等価性 ―― 「未統合」のギャップ(2026-06-16訂正)。** レガシー出力を正解にせず、**保存則・無次元数・解析解・物理的不変量**を等価基準にする手法は、**コード翻訳の外側では確立済み**(メタモルフィックテスト/MR、MMS、計算物理のコード検証)。当初「存在しない」と記したのは誤り。**真の空白は、それらを LLM レガシーマイグレーションの等価性オラクルとして統合した研究が無いこと。** ユーザの「レガシー独自ロジックが数値誤差を生む → 普遍的知見をよりどころに」という発想は、**既存の MT/MMS を移行パイプラインの検証器に据えるという橋渡しで実装可能**(=未踏だが先行分野が支える、実現性の高い提案)。詳細は §2.7 の訂正ノートを参照。
2. **目的言語が Python の研究が手薄。** 本フォルダの実証は C++/Kokkos/Java/Rust が中心。**制御・数値の Python 移行**(NumPy/SciPy 等の数値意味論差、浮動小数点・整数除算・配列添字規則)を正面から扱う等価性研究は空白。
3. **数値許容差の原理的設定。** Fortran→C++ は「4桁丸め」と経験的。**浮動小数点誤差解析に基づく許容差の理論化**(cf. 本テーマ外だが TTrace が分散学習で「数学的解析に基づく許容差設定」を提案している点は方法論的ヒント)。
4. **中間仕様の数値拡張。** Intent-Preserving は「数値規則」を仕様に含めるが浅い。**BSG/中間契約に物理単位・保存則・許容誤差を第一級で組み込み、人間が承認する**構成は提案余地が大きい。

## 7. 研究提案の骨子（上記事実からの統合的示唆）

> 以下は論文事実からの**含意**であり、各論文の主張そのものではない。

- **コア仮説:** 「生成前の中間仕様(契約)+ 検証ループ」という収束パターンを、**Python を目的言語**とし、**数値領域では物理/普遍知見をオラクル**に拡張すれば、レガシー出力をオラクルにできない数値計算でも等価性を担保できる。
- **制御ロジック側:** AgentModernize 流の **BSG(事前/事後/不変条件 + 明示/暗黙ルール区別 + 信頼度・出所つき)** を人間承認ゲート付きで採用。等価性は MatchFixAgent 流の意味解析 + 差分テストで検証。事業価値は**監査証跡**。
- **数値計算側:** 等価性オラクルを2層化 ―(a) レガシー出力との**許容差付き比較**(Fortran→C++ 流)、(b) **物理的不変量/解析解**との照合(新規)。両者が衝突する箇所こそ「レガシー独自の数値誤差」候補として**人間レビューに上げる**(要件再構築の承認点)。
- **評価:** Beyond Translation Accuracy の警告に従い**構成情報を明示した評価**を設計。指標は pass@1・近似機能等価・BER に加え、**物理的整合性指標**(保存則残差等)を新設。MigrationBench のようなベンチ整備も貢献余地。
- **差別化:** ①物理知見オラクル ②Python ターゲット ③数値許容差の原理化 ④中間契約の数値・単位拡張 ―― いずれも本調査では未充足(①は「橋渡し未統合」の意味。§8参照)。

## 8. 橋渡しの土台:普遍知見オラクル(MT / MMS / コード検証)関連文献

§2.7・§6.2 の訂正に伴い、2026-06-16 に追加調査・ダウンロードした「物理/数学的普遍知見をオラクルにする」先行分野の文献。**これらはコード翻訳の文脈の外で確立しており、本研究は『これらをLLMレガシーマイグレーションの等価性検証器として統合する』橋渡しを担う。**

### 8.1 メタモルフィックテスト(MT)/ メタモルフィック関係(MR)

- **Testing Scientific Software: A Systematic Literature Review**(MT_1804.01954, Kanewala & Bieman, 2018)― 62件の系統的レビュー。科学計算ソフトのテスト課題を「(1)ソフト特性に起因(**オラクル問題**)」「(2)科学者と SE の文化差(コードとモデルを不可分視)」に分類。**オラクル問題が中核課題**であることを確立した基礎文献。
- **Metamorphic Relation Generation: State of the Art and Visions**(MT_2406.05397, 2024)― MT は「オラクル問題に対処する主流技術」。MR の系統的生成手法を概観するサーベイ。**研究提案で「どうMRを作るか」の方法論的裏付け**。
- **Towards Automated Metamorphic Test Identification for Ocean System Models**(MT_2103.09782, GEOMAR/Kiel, 2021)― **オラクルが稀な海洋シミュレーションで、既知の物理系の対称性(symmetries)をMRに使う**。MLでMRを自動同定し、**同一海洋モデルの2実装をMRで突き合わせる**(=等価性の差分検証に酷似)。ユーザの「物理の普遍知見をよりどころ」と最も近い実例。重要な限界も明記:**初期出力が誤りでも後続出力がMRの許容変動内なら検出できない**(=レガシー出力をオラクルにできない問題の裏返し)。
- **Testing Causality in Scientific Modelling Software**(MT_2209.00357, Sheffield/Ulm, 2023)― 巨大入力空間・長時間実行・非決定性でテスト困難な科学モデルに対し、**因果推論で既存データから因果効果を推定し、MTのテスト結果を事後的に導く** Causal Testing Framework。3つの実モデルで実証。**高コストな再実行なしに等価性検証する**方向のヒント。
- **Exploring and Lifting the Robustness of LLM-powered APR with Metamorphic Testing**(MT_2410.07516, 2025)― LLMプログラム修復(LAPR)向けMTフレームワーク MT-LAPR。トークン/文/ブロックの3粒度で**9種のMR(意味等価な摂動)**を定義し、平均34.4〜48.5%のテストでLLMの不安定性を露呈。**MTとLLMコードタスクの接続実例**(本研究の隣接技術)。

### 8.2 計算物理のコード検証(Code Verification / MMS)

- **MMS Code Verification of Elastostatic Solid Mechanics in a Commercial FE Solver**(MMS_1902.07608, FDA, 2019)― **Method of Manufactured Solutions(MMS)**で商用FEコード(ABAQUS)をソース改変なしに検証。SymPy/Mathematica で解析的ソース項を生成し、メッシュ細分で2次収束を確認。**MMSの収束次数が軽微なコーディング誤りに感度を持つ**ことを実証。**「正解データ無しに、製造解という普遍的構成でコードの正しさを定量検証」する典型**。医療機器(規制産業)文脈なのもユーザの応用に近い。
- **Numerical scheme ... Swift-Hohenberg ... (MMS order-of-accuracy)**(MMS_2007.16080, 2022)― 非線形PDEの数値スキームを**コード検証で時間・空間2次精度を確認**し、**Lyapunov汎関数の単調減少という物理的性質**をテストケースで確認。**物理不変量を検証オラクルにする**具体例。

### 8.3 精度・要件・形式検証との接続(関連)

- **Methodology for Development of Scientific Software and Test Frameworks in Function of Precision**(SciSW_2203.11650, 博士論文, AGH, 2020)― **「期待結果の精度」を軸にした**科学ソフト開発・テスト枠組み。**数値許容差の原理的設定(§6.2-③)**に直結する方法論。
- **From Scientific Texts to Verifiable Code: Automating the Process with Transformers**(SciSW_2501.05252, KTH, ポジションペーパ, 2025)― **論文中の形式的証明をLLMで検証可能コードに変換**し、責務分離で高レベル骨格→低レベル証明の2段化。**「普遍知見(論文・物理式)から要件・検証コードを再構築し人が承認」**というユーザの第2方針(要件作り直し)に方法論的示唆。

### 8.4 これらを踏まえた「真のギャップ」の再確認
- MT/MR・MMS・コード検証は**いずれもオラクル無しで正しさ/等価性を検証する確立技術**だが、**対象は「単一実装の検証」または「同一言語の2実装比較」**であり、**LLMによるレガシー→現代言語(特にPython)翻訳の等価性判定に組み込んだ研究は本調査範囲に無い**。
- すなわち本研究の核心的貢献は「**新分野の発明**」ではなく「**確立した普遍知見オラクル(MT/MMS)を、LLMマイグレーション・パイプラインの自動検証器+人間承認ゲートとして統合する橋渡し**」と位置づけられる(=実現性・新規性・関連研究の厚みを同時に確保)。
